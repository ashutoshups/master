## Profile photo upload (S3)

This app uploads profile photos directly from the browser to S3 using a **presigned PUT URL** generated by the SSR/Express server.

### Run locally

- Create `.env` from `.env.example` and set:
  - `AWS_REGION`
  - `S3_BUCKET_NAME`
  - and (for local dev) `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`
- Start the backend SSR server (needed for `/api/s3/presign-put`):

```bash
npm run build
npm run serve:ssr:my-app
```

- In another terminal, start Angular (already configured to proxy `/api` to `http://localhost:4000`):

```bash
npm start
```

### Required S3 bucket configuration

1) **CORS** (Bucket → Permissions → CORS). Example for local dev:

```json
[
  {
    "AllowedOrigins": ["http://localhost:4200"],
    "AllowedMethods": ["PUT", "GET", "HEAD"],
    "AllowedHeaders": ["*"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3000
  }
]
```

2) **Public read (only if you want to use the returned `publicUrl`)**

The frontend stores and displays `user.profilePhotoUrl` as a normal `<img src="...">`.
To make that work, the uploaded objects must be readable. A common approach is a bucket policy that allows **GetObject only** for the `profile-photos/` prefix:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowPublicReadProfilePhotos",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::YOUR_BUCKET_NAME/profile-photos/*"
    }
  ]
}
```

If you keep **Block Public Access** enabled, then `publicUrl` will 403; in that case, switch the app to use **signed GET URLs** (presigned reads) or serve through CloudFront.
